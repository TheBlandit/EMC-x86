\section{Introduction}

\subsection{What is the x86 architecture?}

Processors run by executing binary instructions called machine code, this machine code can be made into
a human readable format called assembly which can be turned back into machine code using a program called an assembler.
Each processor has an architecture, this architecture describes how the processor executes the machine code (so different architectures execute code differently,
so have different machine code and assembly languages).
For multiple processors to be able to execute the same machine code they must have the same architecture, some popular architectures include ARM, PowerPC, RISC-V and x86.
x86 is an example of a Complex Instruction Set Computer (CISC), this means that there are lots of instructions
so a lot can be done in a single instruction so is very fast, this however uses a lot of power so is the main architecture in desktop computers.
This is in opposition to ARM which is a Reduced Instruction Set Computer (RISC) which has less instructions so has to execute many to
perform the same task so is slower than x86, this however uses less power so is the main architecture in mobile devices and embedded systems.

\subsection{Basics of x86 assembly}

x86 assembly code contains one instruction per line, the line starts with the name of the instruction followed by its parameters which can either be registers, a memory address or an immediate value (a constant).
Lines and constants can be given labels and can be referred to in instructions.
The assembler only assembles one file at a time so if a label (symbol) is located in another file, the assembler will fail to assemble the program; this is fixed in NASM by marking the symbol in the file its defined in as global (so its accessible to the linker) and as extern in the file its used in.
A program called the linker will combine these object files into a single executable and resolve all extern symbols to their global definitions. Furthermore the linker makes the executable in the correct format for the target platform (eg elf64 on Linux).
Lastly, all code written will be using the Intel syntax using the NASM assembler and will be linked using the ld linker unless stated otherwise.

x86 processors have several different operating modes that they can run in, these operating modes can change the addressing mode, available instruction set and registers.
x86 processors start up in real mode from address FFFF:0000, however the BIOS reads the boot sector and jumps to 0000:7C00 (sometimes 7C00:0000). % TODO: Loading from UEFI

\subsection{Cache}

Cache is a relatively small amount of memory placed within the processor. It is extremely fast SRAM (as opposed to the slower main memory DRAM). There are typically several layers of cache, with each layer getting larger in capacity and slower to access:

\begin{enumerate}
  \item L1d: \\
    Extremely fast data cache
  \item L1i: \\
    Extremely fast instruction cache
  \item L2: \\
    Very fast cache
  \item L3: \\
    Fast cache (shared between cores)
\end{enumerate}

\subsubsection{Cache coherency protocol}
Cache coherency protocols ensure the caches on difference processors [cores] store the most recent, correct copy of data.
x86 uses MSI-based protocols, such as MESI, MOESI and MESIF (with MESI being used in older processors, and the latter two being used in newer AMD and Intel processors respectively);
MSI-based protocols are a type of write-invalidate snooping protocol, this means that a write on one cache invalidates all other cache.
Each cache line is stored with a state, when a cacheline is accessed (read miss or write), its state is updated and it sends a message to all other processors on a bus.
The cache controllers snoops the bus, if a message involves a cacheline in its cache, its state is updated to reflect the message.

The accesses to cache which affect state are:
\begin{enumerate}
  \item PR - current Processor Reads (cache miss)
  \item PW - current Processor Writes
  \item BR - Bus Read (cache miss, another processor reads)
  \item BW - Bus Write (another processor writes)
\end{enumerate}

The state transitions are defined by a finite state machine, the invalid state is considered as a cache miss and uncached items can be considered to be in the invalid state.

\subsubsection{Cache mappings}

Cache mapping describes how the physical address is used to map to the data, there are three types:

\begin{enumerate}
  \item Direct: \\
    Each physical address is split into three parts: tag, block and word.
    The block is used to identify a specific cacheline, each cacheline stores a tag and the tags are compared, if equal its considered a cache hit else it is a cache miss.
    On a hit, the word is used as an offset into the cacheline to retrieve the data.
    On a miss, the new data is fetched from main memory and replaces the cacheline.
  \item Associative: \\
    Each physical address is split into two parts: tag and word.
    Every cacheline is checked in parallel for if the tags are equal.
    On a hit, the word is used as an offset into the cacheline to retrieve the data.
    On a miss, the new data is fetched from main memory and replaces a cacheline usually identified by a LRU algorithm (Least Recently Used).
  \item N-Way Set-Associative: \\
    Each physical address is split into three parts: tag, block and word.
    The block is used to identify a specific set (each set stores N cachelines).
    Each cacheline in the set is checked in parallel for if the tags are equal.
    On a hit, the word is used as an offset into the cacheline to retrieve the data.
    On a miss, the new data is fetched from main memory and replaces a cacheline within the set usually identified by a LRU algorithm.
\end{enumerate}

\begin{table}[H]
  \centering
  \begin{tabular}{|c|p{0.25\linewidth}|p{0.25\linewidth}|}
    \hline \textbf{Mapping} & \textbf{Pros} & \textbf{Cons} \\ \hline
    Direct & Fast (spacial locality) & Cache lines often get evicted if they have the same block \\
    Associative & Cache lines only get evicted if not used (temporal locality) & Slow \\
    Set-Associative & Faster than Associative with similar positives & Slower than Direct and Conflict misses can occur \\ \hline
  \end{tabular}
\end{table}

Most x86 processors use N-Way Set-Associative caches, with the more performant ones using larger set sizes.

\subsubsection{Cache misses}

A cache miss occurs when valid data does not exist in the processor's cache. The first three types of cache miss are commonly known as the three C's.

\begin{enumerate}
  \item Compulsory (cold) miss: \\
    Occurs when accessing memory for the first time (so was never cached)
  \item Capacity miss: \\
    Occurs when cache is full and a cacheline was evicted but later used
  \item Conflict miss: \\
    Occurs when multiple cache lines are mapped to the same location in cache, causing one to get evicted so if read again later it is a miss
  \item Coherence miss: \\
    The (write-invalidate) cache coherency protocol (eg MESI) invalidated the cacheline because it was written in another processor
\end{enumerate}

\subsection{Segmentation}\label{segmentation}
% TODO: types of segment, TTS, and segment registers

Segmentation is a method of memory management that separates parts of a processes address space into different segments (such as code and data), it uses the 6 segment registers to keep track of the segments.
It is used to convert a logical address (which consists of a segment register and an offset) into a linear address.
% Every time a process accesses a memory location, it uses a segment.; make this better
Segmentation has different behaviours depending on what operating mode the processor is in, however unlike paging it is always enabled.

\subsubsection{Real Mode}

The value in the segment register is left shifted by 4 (multiplied by 16) then added to the offset to produce a 20-bit physical address allowing up to 1MiB of memory instead of the 16-bit limit of 64KiB.
The kernel allocates memory for each running process and gives them each different segments, this allows the programs to be able to be loaded into any part of the address space and to not affect other program's memory.
However there is no hardware enforcement of memory protection as any process can read and modify any memory location. % TODO:

\subsubsection{Protected Mode}

% Add flags to selector
The value in the segment register is a segment selector, it is used as an index into the LDT/GDT to select a segment descriptor.
Each segment descriptor contains a base address, limit and flags.
The base address is a linear address and points to the base of the segment, the limit is the size of the segment and the flags set the segments properties such as permissions, 32 or 16 bit code and granularity of the limit.
% add gdt/ldt and their registers and add flags in the segment registers
There are many models which can be used to implement segmentation; some common models are listed below, other models exist and additional functionality can be added to the models such using the code segment to point to ROM.

\begin{enumerate}
  \item Basic Flat Model: \\
    Contains at least two segment descriptors, one for code and one for data (this is because the code descriptor has to be executable so is usually read-only).
    Both descriptors are mapped to the entire linear address space (even if the physical memory does not exist) so it has a base address of 0 and a limit of 4GiB.
    This essentially disables segmentation for both the application and system developer, and since the entire linear address space is mapped no \#GP exceptions will occur.
    Usually implemented alongside paging.
  \item Protected Flat Model: \\
    Similar to the Basic Flat Model but only included the range of addresses for which physical memory actually exists, so the limit is set to the capacity of physical memory.
    If a process accesses a non-existent memory location a \#GP exception will occur.
  \item Multi-Segment Model: \\
    Uses the full capabilities of segmentation to provide hardware enforced protection of memory.
    Each program is given its own segment descriptor table and multiple segments (which may be private or shared with other programs).
    Hardware checks can be used to enforce the program to access memory it owns and in a way the OS allows (such as not writing to code/read-only memory)
\end{enumerate}

\subsubsection{Long Mode}

The behaviour of segmentation in long mode depends if it is running in 64-bit or compatibility mode.
In compatibility mode it functions the same as it does in protected mode.
In 64-bit mode, segmentation is generally disabled (but not completely), which creates a flat 64-bit linear address space.
The base address of the CS, DS, ES and SS are treated as 0 (the effective address is equal to the linear address), and the processor does not perform limit checks (on all segment registers).
The FS and GS registers are often used to reference thread-local or kernel data structures since their base address still applies.

\subsubsection{Segment Selectors}

Segment selectors are used as a 16-bit identifier for a segment descriptor.
Segment selectors contain 3 parts:

\begin{enumerate}
  \item Bits 3:15 - Index: \\
    The index into the descriptor table, selects one of the 8192 segment descriptors in the table.
    Its value is multiplied by 8 then added to the base address of the of the descriptor table.
  \item Bit 2 - Table Indicator (TI) flag: \\
    Selects between the two descriptor tables: 0 selects the GDT to be used, 1 selects the LDT to be used.
  \item Bits 0:1 - Requested Privilege Level (RPL): \\
    Specifies the privilege level of the selector. %, 0 being the most privileged (supervisor) and 3 being the least (user).
\end{enumerate}

If the index is 0, it implicitly selects a null segment.
Additionally both the CS and SS registers have to be loaded with a present descriptor else a \#GP exception is generated; furthermore if the SS is loaded with a non-writeable segment, a \#GP is generated.

\subsubsection{Segment descriptors}

Segment descriptors are identifiers for segments of memory.

Many of the bits are described below:

\begin{enumerate}
  \item E = Expansion-direction: \\
    If 0, the segment is expand-up;
    this means that the range of valid offsets is [0, limit].
    Therefore the range of linear addresses of the segment is [base, base + limit], increasing the limit increases the maximum address (useful for data segments).
    If 1, the segment is expand-down;
    this means that the range of valid offsets is (limit, 0xFFFFF] if B = 0 or (limit, 0xFFFFFFFF] if B = 1.
    Therefore the range of linear addresses of the segment is (base + limit, base + (0xFFFFF or 0xFFFFFFFF)], decreases the limit decreases the minimum address (useful for stack segments).
  \item W = Write-enabled: \\
    Allows the segment to be written; the stack segment must be writeable, loading the SS register with a read only segment causes a \#GP exception.
  \item A = Accessed: \\
    The processor sets it to 1 if it accessed the segment.
    The kernel can clear its value and it is often useful for debugging.
    If the segment descriptor is placed in ROM, the processor may enter an infinite loop attempting to set the Accessed bit.
    To prevent this, the segment descriptors in ROM should have their Accessed bit set.
  \item C = Conforming: \\
    A transfer of execution into a more privileged conforming segment will continue at the current privilege level.
    Transferring execution into a more privileged non-conforming segment or less privileged segment will result in a \#GP exception if a call or task gate is not used. % TODO: Check that a high to low privilege will prevent a GP if a gate is used
    All data segments are non-conforming so cannot be accessed by less privileged processes, however unlike code segments higher privileged processes can access lower privileged data segments without the use of a gate.
  \item R = Read-enabled: \\
    Allows the code segment to be read and is useful if constants or static data are placed within the code segment.
    The code segments can be read either by using a CS override or by loading the segment into a data-segment register (DS, ES, GS, FS).
\end{enumerate}

\subsubsection{GDT - Global Descriptor Table}

Contains the segment descriptors, its base address (linear) and limit (number of bytes in the GDT - 1) are stored in the GDTR (GDT Register).
The \texttt{LGDT} instruction stores the base address and limit from a memory location into the GDTR.
The $0^{th}$ descriptor of the GDT is often used to store the address and limit since it will never be loaded as a descriptor since the selector would be null.

\subsection{Paging}

Paging is a method of memory management that separates a programs address space (linear address) from the physical address space.
It occurs after segmentation but unlike segmentation it does not need to be enabled, if it is disabled linear addresses are the same as physical addresses.
The OS splits up the physical address space into 4KiB pages, each process has a paging structure which the OS manages, it consists of many layers;
each layer consists of the entries for either the physical address of the next layer or page and flags.
When a process accesses memory, the processor hardware converts the linear address into its corresponding physical address using the paging structure (by traversing the layers) and then uses it to access memory, this is called a hardware-assisted page walk.

Each structure is 4096 bytes in size and comprises a number of individual entries.
In 32-bit paging each entry is 32 bits (4 bytes) so there are 1024 entries in each structure.
In 64-bit paging each entry is 64 bits (8 bytes) there are thus 512 entries in each structure.
(In PAE however there is a structure 32 bytes in size containing 4 64-bit entries)

The processor uses the upper portion of a linear address to identify a series of paging-structure entries. The last of these entries identifies the physical address of the region to which the linear address translates (called the page frame). The lower portion of the linear address (called the page offset) identifies the specific address within that region to which the linear address translates.

Each paging structure entry contains a physical address - either the address of another paging structure or a page frame. In the first case it is said to reference the other paging structure; in the second the entry is said to map a page.

CR3 contains the first paging structure. (unless HLAT paging)

A paging-structure entry is marked as not present if its P flag (bit 0) is clear, if it is encountered during translation it causes a page-fault exception.

Usually an address is mapped to a 4 KiB page, if it is not the final paging structure then bit 7 (PS - page size) is checked, if it is 0 the process continues normally, else if it is 1 the entry maps a larger page (4MiB for 32-bit paging if CR4.PSE = 1, 2MiB for other paging modes regardless of CR4.PSE; 4-level and 5-level paging may also support 1GiB pages).
Bit 7 is used for other purposes if its in the final paging structure.

\subsection{Memory addressing}

Memory addressing is used to access specific locations in main memory, memory addresses have the syntax: \\
\texttt{[Segment:Base + Index*Scale + Displacement]} \\
x86 has multiple different types of address:

\begin{enumerate}
  \item Logical: \\
    Consists of a segment selector (16-bit) and an offset (same bit size as the code).
    Usually inferred from the type of memory access (eg uses CS for code, SS for stack) or from a segment override.
    Produces a linear address after segmentation.
  \item Effective: \\
    Consists of a combination of a \texttt{Base} register, an \texttt{Index} register (optionally with an immediate \texttt{Scale} which is 1, 2, 4, or 8) and an immediate \texttt{Displacement}.
    Once the effective address is computed it is used as the offset in the logical address.
  \item Linear: \\
    If paging is not used, the linear address is equal to the physical address.
    If paging is used, the corresponding physical address is found using the paging hierarchy.
  \item Virtual: \\
    A valid linear address that does not correspond to an in-memory page, causing the kernel to generate the page via an interrupt (eg load it from disk).
  \item Physical: \\
    The physical location of the memory in main memory.
\end{enumerate}

\section{Instructions}

The format of the instruction is \texttt{instruction op1, op2, op3} eg \texttt{'MOV RAX, 10'}.
Each instruction has different limitations to what type an operand can be, these types are

\begin{table}[H]
  \centering
  \begin{tabular}{|c|c|l|}
    \hline \textbf{Symbol} & \textbf{Name} & \textbf{Description} \\ \hline
    r & General purpose register & A register such as eax \\
    m & Memory address & A memory address such as \texttt{[eax + 4*ebx + 4]} \\
    i & Immediate & A constant value such as 80h or a pointer to a label \\
    s & Segment Register & A segment register such as es \\ \hline
  \end{tabular}
\end{table}

The types of an operand or identical instructions with different names will be separated by \texttt{(|)}, the size in bits may also be appended to the type and (') may be used to exclude a single size (eg \texttt{i8|i16} means an 8 or 16 bit immediate value and \texttt{r'8} means a 16, 32 or 64 bit register).
Some instructions have more than one row of operands, each row shows a different combination of operand types. Most instructions should only operate on a single word size (eg 64-bit), if an instruction is used with different sized operands then it may cause an assembler error, cause an invalid opcode exception, truncate the source or bit extend it (either arithmetically or logically).
Furthermore many of these instructions set/clear flags, for example the OF (overflow flag) is set on signed overflow, CF (carry flag) is set on unsigned overflow and ZF (zero flag) is set when the result is zero.
Some instructions can be prefixed with LOCK, this makes the instruction atomic and will give the thread exclusive access to the cache line it is accessing; LOCK can only be used when the destination is a memory location and the instruction supports it else it will raise a UD exception.

\section{Interrupts}

% https://wiki.osdev.org/Interrupts
Interrupts are signals from devices to the CPU which makes it immediately pause execution and run code to handle the interrupt.
When an interrupt occurs, the CPU finds its entry in an OS provided lookup table called the Interrupt Descriptor Table (IDT) and can have up to 256 entries;
once the CPU locates the entry, it jumps to the code to handle the interrupt which is called the Interrupt Service Routine (ISR) or interrupt handler.

There are three types of interrupts:
\begin{enumerate}
  \item Exceptions: Generated internally by the CPU and are used to alert the kernel of an event which requires its attention such as a Double Fault, Page Fault, General Protection Fault, etc

\end{enumerate}
